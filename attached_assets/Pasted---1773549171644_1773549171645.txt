开始设计并开发 周报系统。这不是单独一个页面，而是一套完整的“学生提交 + 导师管理”的周报系统。
请一次性从系统角度设计并实现，不要把学生端和导师端拆成两套互相独立的逻辑。
一、系统目标
周报系统包含两个主要入口：
1. 学生端
位置：个人 > 我的周报
学生在这里：
新建周报
保存草稿
提交周报
查看历史周报
查看导师反馈
修改后重新提交
2. 导师端
位置：团队 > Reports
导师在这里：
查看全组周报
查看谁已提交 / 谁未提交
查看周报详情
给反馈
标记待修改 / 已批阅 / 风险项
这两个页面读写同一套周报数据，不允许做成两套分离系统。
二、统一数据模型与状态流转
1. 周报核心状态
请统一使用这一套状态：
draft 草稿
submitted 已提交
under_review 待批阅
needs_revision 待修改
reviewed 已批阅
导师端还需要一个计算态：
missing 未提交（不是数据库真实记录，而是团队视角计算出来的占位项）
2. 周报内容结构
周报正文请采用结构化内容，不要只用一个大文本框：
标题
周次范围
本周完成内容
当前实验进展
遇到的问题
下周计划
需要导师帮助的事项
关联实验记录
附件
3. 导师反馈
周报应支持导师评论 / 反馈，并允许学生在 needs_revision 状态下修改后重新提交。
三、学生端：个人 > 我的周报
页面目标
这是学生自己的周报工作区，不是导师管理页。
页面布局
建议使用：
顶部工具栏
左侧我的周报列表
右侧周报编辑 / 详情区
顶部工具栏
包含：
页面标题：我的周报
当前周信息
主按钮：新建本周周报
辅助按钮：查看历史周报
左侧周报列表
每条显示：
周报标题
周次范围
最后更新时间
状态标签
是否有导师反馈
右侧区域
根据状态分为：
编辑态（draft / needs_revision）
只读态（submitted / reviewed）
学生操作
支持：
新建本周周报
保存草稿
提交周报
查看导师反馈
修改后重新提交
关联实验记录
上传附件
四、导师端：团队 > Reports
页面目标
这是导师视角的周报管理工作台。
页面布局
建议使用：
顶部工具栏
左侧周报列表
右侧详情面板
顶部工具栏
包含：
周选择器（本周 / 上周 / 自定义）
搜索框
状态筛选
统计信息（应交、已交、未交、待批阅）
左侧列表
每条显示：
学生头像
学生姓名
周报标题
周次
提交时间
状态标签
风险标签（可选）
一行摘要
注意：
未提交学生也要显示为占位项，例如：
刘浩 — 本周未提交
右侧详情面板
显示：
周报头部信息
本周完成内容
当前实验进展
遇到的问题
下周计划
需要导师帮助
关联实验记录
导师反馈区
导师操作
支持：
写反馈评论
标记待修改
标记已批阅
标记风险
查看未提交情况

六、前端开发要求
这不是一次性演示代码，而是正式产品代码。请按长期维护的项目结构实现，不要写成 demo 式堆叠代码。
要求：
页面层、组件层、业务逻辑层、数据访问层分开
不要把页面UI、状态管理、接口请求、类型定义写在同一个文件里
可复用组件单独拆分
类型定义单独管理
API 请求单独封装
左侧导航、表单、卡片等公共模块优先复用
不要制造超长文件，单个文件尽量职责单一
后续功能会持续扩展，请按可维护项目结构来写
先实现 MVP 功能，再顺手整理结构。不要留下后期难以维护的大文件和耦合代码。

请按这个顺序推进：
Phase 1：数据模型与接口骨架
先把 weekly_reports / comments / 状态流转设计清楚

先让学生能：

再补：
