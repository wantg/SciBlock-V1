现在开始 Phase 3：实验报告内容质量优化。

前提说明：
Phase 2 和 Phase 2.1 已经把实验报告的基础链路打通了：
- 报告可持久化
- AI 只生成 3 个核心区块（实验概述 / 结果分析 / 实验结论）
- 手动保存已可靠
- regenerate 已无 race condition
- source 状态机已可区分 ai / ai_modified / manual / stub

所以这次不要再扩展新功能，不要做版本历史、PDF 导出、区块级重生成等。  
这次只聚焦一件事：

把 AI 生成的三个区块，从“能输出文字”优化成“更像正式实验报告、保守可信、对用户有实际帮助”的内容。

一、Phase 3 总目标

把当前 AI 生成的三块内容：

- 实验概述
- 结果分析
- 实验结论

从目前可能存在的：
- 空泛
- 套话
- 只是在复述输入
- 对 objective 没闭环
- 数据不足时表达不分级

优化为：
- 有明确信息层级
- 有保守表达边界
- 能根据数据充分程度输出不同强度的表述
- 结论能明确回应实验目标
- 尽量减少空话和假分析

二、这次必须先做代码审查，再实施

请先基于当前真实代码，检查以下问题是否存在，并在实施时一起解决：

1. 当前 prompt 是否只看 report model，不看任何用户自由文本
2. 当前 summary / analysis / conclusion 是否是一次请求生成
3. 当前系统 prompt / user prompt 是否对“禁止编造”约束还不够细
4. 当前 renderer 是否无法区分“明确结论 / 初步判断 / 信息不足”
5. 当前 report model 是否缺少足够支持 AI 判断的数据上下文
6. 当前输入里是否遗漏了：
   - 用户自由补充的文字
   - 模块里的 notes / description
   - data / measurement 中有价值但目前未进入 prompt 的字段

三、重点优化目标

1. 建立“信息充分度分级”机制
AI 不能只会两种状态：
- 正常生成
- “暂无数据支撑”

请把输出逻辑明确分成至少 3 档：

A. 信息较充分
适用于：
- 已有较明确的测量方法
- 已有数据类型
- 目标相对明确
输出应表现为：
- 可以形成较明确的实验概述
- 可以给出初步分析
- 可以形成阶段性结论
- 但仍不得虚构具体结果数值或趋势

B. 信息部分充分
适用于：
- 有实验目标和过程
- 有部分测量方法或数据类型
- 但对结果支持有限
输出应表现为：
- 可以描述已完成的实验过程和获取的数据类型
- 分析必须明确指出“当前支持有限”“仅能做初步判断”
- 结论必须明确标注“阶段性”或“有待进一步验证”

C. 信息明显不足
适用于：
- 缺少测量信息
- 缺少数据类型
- 缺少可支撑分析的记录
输出应表现为：
- 明确说明当前报告只能做记录性总结
- 不能硬写分析
- 应提示需要补充哪些关键信息后才能形成更可靠结论

要求：
- 这三档不要只存在于 prompt 文案里
- 尽量在后端生成逻辑中显式判断，或者在输入给模型时带上一个 adequacy / evidenceLevel / completenessLevel 之类的字段
- 不要把一切都交给模型自己猜

四、实验概述（summary）优化要求

当前 summary 不能只是把已有摘要再换一种说法。请优化为：

1. 要有“做了什么 + 为什么做 + 用什么方式获取信息”的结构
至少应覆盖：
- 本次实验面向什么对象/系统
- 目标是什么
- 采用了哪些关键步骤或表征方式
- 当前记录能支持到什么层次的认识

2. summary 要更像“报告开头摘要”，而不是“字段复述”
避免这种感觉：
- 机械重复 title / objective / N 个对象 / M 个步骤
应更接近：
- 一段连贯的实验说明

3. 当信息不足时，summary 也要保守
不要为了像报告而假装已经获得明确发现。

五、结果分析（analysis）优化要求

这是本次最关键区块。

请让 analysis 体现下面几个维度：

1. 分析不是复述过程
不能把“做了哪些步骤、用了哪些仪器”重新说一遍就叫分析。

2. 分析必须尽量围绕“数据/测量/目标”的关系来写
至少尝试表达：
- 当前记录中有哪些测量或数据获取手段
- 这些手段理论上对应什么观察目标
- 基于现有记录，能支持什么程度的判断
- 哪些地方证据还不够

3. 必须显式包含“不确定性/局限性”表达
当数据不足时，要能自然表达：
- 当前仅能做初步分析
- 尚不能支持明确判断
- 缺乏对照、结果值、异常说明或进一步表征信息

4. 禁止伪分析
例如不能出现这种问题：
- 没有具体结果，却说“结果表明样品性能优异”
- 没有趋势数据，却说“呈现明显上升趋势”
- 没有对照，却说“优于对照组”

请明确增强 prompt 和后端保护逻辑，避免这种生成。

六、实验结论（conclusion）优化要求

结论必须和 objective 闭环，这是这次的核心要求之一。

请确保 conclusion 尽量回答下面问题：

1. 本次实验是否对实验目标提供了支持
2. 支持程度如何
   - 明确支持
   - 初步支持
   - 暂不足以判断
3. 当前结论是否需要更多数据/对照/重复实验来验证

要求：
- 结论不要只是再总结一次 summary
- 结论应带有判断层级
- 但判断必须受证据约束
- 当证据不足时，要明确说“不足以形成明确结论”，而不是硬写正向结论

七、把用户自由文本和更多上下文纳入 AI 输入

当前请重点检查并补充以下输入来源，凡是已有真实字段且适合纳入 prompt 的，请尽量纳入：

1. 模块级 description / notes / remarks / extra text
2. 用户在实验记录中手动补充的说明性文字
3. measurement 的 conditions
4. data 的 attributes 中除“单位”之外对判断有帮助的信息
5. operation steps 的 notes 或更多关键参数
6. system/preparation 中对实验理解有帮助的文本字段

要求：
- 不是无脑全塞给模型
- 而是做一层筛选和整理，进入更适合 prompt 的 context 字段
- 如果字段噪声太大，请你自己做合理裁剪和归纳

八、Prompt 设计要求

请重构 prompt，但保持“只输出结构化 JSON”的方式，不要让模型直接输出 HTML。

建议返回结构至少包含：
- summary
- analysis
- conclusion
- confidenceLevel 或 evidenceLevel（可选，但推荐）
- missingInfoHints（可选，但推荐）

要求：
1. system prompt 要更明确约束：
- 不得编造实验结果
- 不得编造数值、趋势、对照结论
- 不得把实验目的当作实验结果
- 必须区分“已有记录支持”和“推测/待验证”

2. user prompt 要包含：
- 结构化 report model
- completeness / evidence 判断结果
- 可用的自由文本上下文
- 明确的写作要求（实验报告风格，不夸张，不营销）

3. 如果返回 JSON 校验失败，fallback 仍要保留，但 fallback 文案也请优化，不要太僵硬

九、后端保护逻辑要求

这次不要只靠 prompt 自律。请在后端增加一层轻量保护逻辑，例如：

1. 当 measurement/data 信息严重不足时
- 给 AI 明确的 low-evidence 标记
- 或者直接限制 analysis / conclusion 的语气模板

2. 当 objective 缺失或非常短时
- 结论不要强行写闭环判断
- 应切换为“当前实验目标描述不足，建议补充后再生成结论”

3. 如果 AI 返回内容明显越界
例如出现：
- 虚构数值
- 明显的结果性断言但输入无支撑
你可以考虑增加轻量校验/拦截/替换策略
不要求复杂 NLP 检测，但要有基本保护意识

十、UI 层的小幅增强（只做轻量，不做大改版）

如果你认为合适，请轻量增加以下能力之一或几个：

1. 在报告中对 analysis / conclusion 的语气做更清晰区分
例如：
- 初步分析
- 阶段性结论
- 信息不足提示

2. 如果后端返回 evidenceLevel / confidenceLevel
可在 UI 里做一个很克制的小标签显示，例如：
- 信息充分
- 信息有限
- 信息不足

要求：
- 轻量
- 不影响现有报告结构
- 不要做复杂设计改版

十一、这次不要做的事

本轮不要做：
- 报告版本历史
- PDF / Word 导出
- 局部区块重生成
- 多模型切换
- 导师批注实验报告
- 自动保存 typing
- 差异对比视图
