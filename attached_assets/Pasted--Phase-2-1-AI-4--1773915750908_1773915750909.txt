先暂停新功能扩展，先做 Phase 2.1 稳定性修复。这一轮目标不是增强 AI，而是把实验报告功能从“可演示”提升到“可日常使用的可靠功能”。

请优先修复下面 4 个问题，按顺序实施。

一、修复“保存修改”假保存问题（最高优先级）

这是当前最严重的问题。用户点击“保存修改”后，不能再只是本地更新 + 2 秒后异步 PUT。对于实验报告这种正式内容，“点击保存”必须语义上等于“已经成功保存到后端”，否则属于严重产品缺陷。

请这样修改：

1. handleSave() 改为真正 await 保存
- 点击“保存修改”后，直接调用后端 PUT /api/experiments/:id/report
- 必须 await 请求完成
- 成功后再退出编辑模式
- 失败则保留编辑状态，并给出明确错误提示

2. 去掉这个场景下的 debounce
- updateReport() 当前的 2 秒 debounce 不适用于“点击保存”场景
- 对按钮保存来说，应当立即保存
- 如果你想保留 debounce，请仅用于未来真正的自动保存 typing 场景；当前按钮保存不能 debounce

3. 增加保存中状态
- 保存按钮在请求中应进入 loading / disabled 状态
- 防止用户重复点击
- UI 上给出“保存中 / 已保存 / 保存失败”的明确反馈

4. 刷新后验证
- 修复后请验证：点击保存成功后，立即刷新页面，报告内容仍然存在且为最新版本

二、修复“重新生成”竞态问题

当前 handleRegenerate() 里 clearReport() 和 triggerReportGeneration() 几乎同时执行，属于 race condition。不能继续依赖“DELETE 比 AI 快”这种偶然顺序。

请做以下任一方案，并说明你选了哪个：

方案 A（更推荐）：
新增后端 regenerate 语义，原子执行“覆盖旧报告并重新生成”
例如：
POST /api/experiments/:id/report/regenerate
由后端统一完成覆盖与生成，前端不再自己 clear + generate 串联

方案 B（过渡方案）：
前端严格串行：
- await clearReport()
- clear 成功后再 await generateReport()

无论选哪种方案，目标都是：
“重新生成”不能再依赖两个并发请求的自然排序。

三、收口 source 语义

当前 source 无法区分“AI 生成后又被人工修改”的状态，这会影响后续审计、版本对比和产品语义。

请升级 source 语义，至少支持：

- "ai"：纯 AI 生成且未人工修改
- "manual"：纯手工保存/编辑生成
- "ai_modified"：AI 生成后被用户手动修改过

并明确规则：
- AI 成功生成 → ai
- 用户手动保存 AI 稿件 → ai_modified
- 用户从空白开始写并保存 → manual
- clear → null

如果你判断需要增加额外字段，而不是只扩展 source，也请说明原因和方案。

四、明确 report_model_json 的产品语义

当前 report_model_json 在手动编辑后与 report_html 永久脱钩。请不要隐性维持这种“不一致但没人知道”的状态。

请明确做出一个短期方案：

方案建议：
- 当前将 report_model_json 定义为“最近一次生成时的报告模型快照 / 审计快照”
- 手动编辑 report_html 时，不强行同步改写 report_model_json
- 在代码注释和文档中明确：手动编辑后，report_model_json 不再代表最终渲染内容，只用于调试/审计
- 不要在任何生产逻辑里把它当作最终 truth source

如果你认为更好的方案是：
- 手动保存时也更新 report_model_json
请你说明如何更新，以及是否会丢失 AI/规则生成阶段的快照价值
